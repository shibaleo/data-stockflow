# ドメイン基本構成要素の整理

## 前提：システムの位置づけ

本システムは**複数の業務アプリが共有する汎用基盤**である。

- financial book を使う財務会計アプリ
- stock book を使う在庫・残高管理アプリ
- project を使うプロジェクト管理アプリ
- …etc.

これらの**隣接ドメイン**が同じ基盤を共有するため、基盤はアプリ間で解釈がブレない共通の語彙と構造を提供する必要がある。

## 線引きの基準

| 基準 | 層 | 例 |
|---|---|---|
| **ないとデータの整合性が壊れる** | 基盤 | 貸借均衡、ハッシュチェーン、append-only |
| **ないとアプリ間で解釈がブレる** | 基盤 | 精度、丸め、正常残高、期間の包含判定 |
| **特定の業務ドメインに属する** | アプリ | 繰越処理、承認フロー、税計算、原価計算 |

## 6区分マトリクス

各エンティティの概念を以下の6区分で整理する：

|  | インフラ（基盤） | アプリ層 |
|---|---|---|
| **現状** | 実装済み | 実装済み or 未実装だがアプリの責務 |
| **過剰** | 基盤にあるがアプリに移すべき | — |
| **追加** | 基盤に追加すべき | 将来アプリで実装すべき |

## Period（期間）

|  | インフラ | アプリ |
|---|---|---|
| **現状** | コード、開始日、終了日、ステータス（open/closed）、親子階層、close/reopen | — |
| **過剰** | — | — |
| **追加** | 区切り (Cutoff) — 期間を2つの区切りの間として構造的に定義 | 繰越処理 — 期末残高→期首残高の仕訳生成 |
|  | 連続性検証 — 期間の隙間・重複がないことを保証 | 決算整理仕訳の生成 |
|  | 包含判定 — 日付→期間の問い合わせ API | finalize のタイミング判断 |
|  | finalize — closed→finalized の不可逆遷移 + 書き込み拒否 | |

> **隣接ドメイン境界**: 財務会計は月次/年次で区切り、在庫管理は日次で区切る。基盤が「区切り」と「包含判定」を持つことで、どのアプリも同じ期間構造の上で動ける。

## Book（帳簿）

|  | インフラ | アプリ |
|---|---|---|
| **現状** | コード、名前、通貨単位（unit/symbol/position）、type_labels、CRUD + 無効化 | — |
| **過剰** | — | — |
| **追加** | 精度 (Precision) — 帳簿ごとの小数桁数 + amount の桁数検証 | 通貨換算 |
|  | 丸め規則 (Rounding) — 端数処理方法の定義 | 帳簿間の連結・消去 |

> **隣接ドメイン境界**: financial book は `JPY` で小数0桁、stock book は `kg` で小数3桁。帳簿が精度を宣言することで、どのアプリが仕訳を作っても同じ桁数で記録される。

## Account（勘定科目）

|  | インフラ | アプリ |
|---|---|---|
| **現状** | コード、名前、account_type（5種）、sign（派生）、親子階層、CRUD + 無効化 | — |
| **過剰** | — | — |
| **追加** | 正常残高 (Normal Balance) — debit/credit の明示的属性 | 異常残高の検知・警告 |
|  | 集計/明細区分 (Summary/Detail) — 集計科目への直接仕訳を拒否 | 勘定体系の設計 |
|  | | 科目グループ（BS/PL 表示区分）— 階層で表現可能 |

> **隣接ドメイン境界**: 財務アプリが「現金」科目を asset/debit として扱い、在庫アプリも同じ科目を触る。正常残高と集計/明細区分を基盤が持つことで、科目の意味がアプリ間で統一される。

## JournalLine（仕訳行）

|  | インフラ | アプリ |
|---|---|---|
| **現状** | side、amount（正数→内部符号化）、account_key、department_key、counterparty_key、description | — |
| **過剰** | — | — |
| **追加** | 数量 (Quantity) — nullable な物量フィールド | 数量 × 単価 = 金額 の計算ロジック |
|  | 単価 (Unit Price) — nullable な単価フィールド | 原価計算（先入先出、移動平均等） |
|  | 補助通貨金額 (Secondary Amount) — 外貨/別単位の元金額 | 為替レート管理、換算処理 |
|  | | 税区分・税計算 |

> **隣接ドメイン境界**: 在庫アプリは「100個 × @50円 = 5,000円」を記録し、財務アプリは金額だけ見る。数量・単価を基盤の行構造に持つことで、どちらのアプリも同じ仕訳行を共有できる。

## まとめ

### 過剰なもの

なし。現状の基盤に不要なものは含まれていない。

### 基盤に追加すべきもの

いずれも「ないとアプリ間で解釈がブレる」もの。

| エンティティ | 概念 | 隣接ドメイン境界 |
|---|---|---|
| **Period** | 区切り、連続性検証、包含判定、finalize | 財務=月次、在庫=日次 が同じ期間構造を共有 |
| **Book** | 精度、丸め規則 | JPY 帳簿と kg 帳簿が同じ検証ルールで動く |
| **Account** | 正常残高、集計/明細区分 | 複数アプリが同じ科目を統一的に扱える |
| **Voucher** | 添付参照 | どのアプリの証憑も同じ仕組みで紐づく |
| **JournalLine** | 数量、単価、補助通貨金額 | 在庫アプリと財務アプリが同じ行構造を共有 |

### アプリ層で実装すべきもの

特定の業務ドメインに属するもの。

- **財務会計**: 繰越、決算整理、税計算、通貨換算、連結
- **在庫管理**: 原価計算、入出庫ロジック
- **プロジェクト管理**: 予算、予実対比、進行管理
- **共通業務**: 承認フロー、配賦、定型仕訳、採番規則
