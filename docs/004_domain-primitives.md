# ドメイン基本構成要素の整理

各ドメインエンティティが本来持つべき基本概念（プリミティブ）を洗い出し、現状の実装と対比する。
「自動○○」のような高レベル機能ではなく、それらを構成するアトミックな概念を対象とする。

---

## Period（期間）

### 現状
- コード、開始日、終了日、ステータス（open / closed）、親子関係
- close / reopen 操作

### 足りない基本概念

| 概念 | 説明 | 現状 |
|---|---|---|
| **区切り (Cutoff)** | ある時点を境に「ここまで」と定める行為。期間は2つの区切りの間として定義される | なし。start_date / end_date は単なる日付フィールド |
| **繰越 (Carry-forward)** | 区切り時点の残高を記録し、次の期間の期首残高とする行為 | なし |
| **期首残高 (Opening Balance)** | 期間の開始時点における各勘定の残高 | なし。レポートは全期間通算で計算 |
| **期末残高 (Closing Balance)** | 区切り確定時に記録される各勘定の残高スナップショット | なし |
| **連続性 (Continuity)** | 区切りが隙間なく並んでいること。前の期間の end と次の期間の start が連続する | 検証なし。任意の日付で作成可能 |
| **包含判定 (Containment)** | ある日付がどの期間に属するかを判定する | なし。Voucher の period_id は手動指定 |
| **確定 (Finalization)** | closed の先の最終確定。reopen 不可にする不可逆な状態遷移 | status に "finalized" は定義済みだが操作なし |

## Book（帳簿）

### 現状
- コード、名前、通貨単位（unit / symbol / position）、勘定科目ラベル（type_labels）
- CRUD + 無効化

### 足りない基本概念

| 概念 | 説明 | 現状 |
|---|---|---|
| **精度 (Precision)** | 金額の小数点以下桁数。通貨なら 0 or 2、重量なら 3 など | なし。amount は DECIMAL(15,0) 固定 |
| **丸め規則 (Rounding)** | 端数処理の方法（四捨五入、切り捨て、銀行丸めなど） | なし |
| **基準通貨との換算 (Exchange)** | 複数帳簿間で金額を比較するための換算レート | なし。帳簿間の関係が定義されていない |
| **帳簿間関係 (Inter-book Relation)** | 連結・内部取引消去など、帳簿同士がどう関係するか | なし |

---

## Account（勘定科目）

### 現状
- コード、名前、account_type（5種）、sign（派生）、親子関係
- CRUD + 無効化

### 足りない基本概念

| 概念 | 説明 | 現状 |
|---|---|---|
| **正常残高 (Normal Balance)** | その科目が通常持つべき残高の方向（debit / credit） | sign で暗黙的に表現しているが、明示的なフィールドではない |
| **集計科目 / 明細科目 (Summary / Detail)** | 親科目は配下の集計用で直接仕訳不可、末端科目のみ仕訳可能 | 区別なし。親科目にも仕訳できる |
| **科目グループ (Account Group)** | BS / PL の表示区分（流動資産、固定資産、売上原価など） | account_type の5種のみ。中間分類がない |
| **使用制約 (Usage Constraint)** | 特定の仕訳種別や部門でのみ使用可能といった制約 | なし |

## Voucher（伝票）

### 現状
- idempotency_key、voucher_code、posted_date、description、source_system
- sequence_no + header_hash によるチェーン
- 作成・一覧・詳細（更新不可）

### 足りない基本概念

| 概念 | 説明 | 現状 |
|---|---|---|
| **承認 (Approval)** | 伝票が正式に記帳される前の承認状態 | なし。作成即確定 |
| **下書き (Draft)** | 未確定の状態。承認前に編集可能 | なし |
| **添付 (Attachment)** | 証憑（領収書、請求書など）との紐付け | なし |
| **取消 (Void)** | 伝票全体を無効にする操作（逆仕訳とは別に、伝票レベルで「なかったことにする」） | なし。Journal 単位の逆仕訳のみ |

---

## Journal（仕訳）

### 現状
- book_key、journal_type_key、voucher_type_key、project_key、adjustment_flag
- 作成・更新（新リビジョン）・無効化・逆仕訳・履歴

### 足りない基本概念

| 概念 | 説明 | 現状 |
|---|---|---|
| **貸借均衡 (Balance Assertion)** | 借方合計 = 貸方合計の検証。概念としては存在するが明示的なドメイン表現がない | routes/vouchers.ts 内のバリデーションとして実装。ドメイン層としては未分離 |
| **配賦 (Allocation)** | 1つの金額を複数の科目・部門に按分する行為 | なし。手動で複数行を作成するしかない |
| **定型仕訳 (Recurring Template)** | 繰り返し発生する仕訳のテンプレート | なし |

---

## JournalLine（仕訳行）

### 現状
- side（debit/credit）、amount（正数→内部符号化）、account_key、department_key、counterparty_key
- Journal と一体で作成・更新

### 足りない基本概念

| 概念 | 説明 | 現状 |
|---|---|---|
| **数量 (Quantity)** | 金額とは別の物量。在庫管理や原価計算で「100個 × @50円 = 5,000円」のように使う | なし |
| **単価 (Unit Price)** | 数量と組み合わせて金額を算出する基礎 | なし |
| **税区分 (Tax Classification)** | 課税・非課税・免税・不課税の区分。消費税計算の基礎 | なし |
| **機能通貨金額 (Functional Currency Amount)** | 外貨建て取引の場合の機能通貨（基準通貨）換算額 | なし |

## 優先度の考え方

上記の概念をすべて実装する必要はない。以下の基準で優先度を判断する：

1. **他の機能の前提となるもの** — Period の区切り・繰越・包含判定。これがないと期間ベースの集計が正しく機能しない
2. **データの正確性に直結するもの** — Book の精度・丸め、Account の集計/明細区分、JournalLine の貸借均衡のドメイン分離
3. **業務フローを成立させるもの** — Voucher の承認/下書き、Counterparty の取引先区分
4. **利便性・分析精度の向上** — 配賦、定型仕訳、予算、税区分など
